Custom CUDA Library for CNN Pre-Processing

Hand-optimized, shared-memory tiled CUDA kernels that accelerate CNN preprocessing - packaged so any Python ML pipeline can use them without a rewrite.

The problem

Data preprocessing - matrix multiplication and image convolution - is a hidden bottleneck in CNN training. Off-the-shelf CPU paths waste GPU time, and rewriting an entire pipeline in C++ to fix it isn't practical.

What I built

  • Matrix multiplication, four ways: a CPU baseline, a naive CUDA kernel (one thread per output element), a shared-memory tiled kernel (16x16 tiles), and a cuBLAS (cublasSgemm) reference - all benchmarked across N = 64 to 2048.
  • Image convolution on the GPU: edge-detection, Gaussian-blur, and sharpen filters at 3x3, 5x5, and 7x7, with one CUDA thread computing each output pixel.
  • Python-bindable .so libraries: wrapped the kernels behind extern "C" entry points (nvcc -Xcompiler -fPIC -shared) and loaded them from Python with ctypes over NumPy float32 arrays - GPU acceleration with no pipeline rewrite.
  • Timed precisely with CUDA events (with warm-up launches) on an NVIDIA Tesla T4 (-arch=sm_75).

How it works

  • Tiling for memory reuse: each block loads a 16x16 tile of A and B into on-chip shared memory once, cutting the redundant global-memory traffic that bottlenecks the naive kernel.
  • Drop-in Python interface: ctypes declares the argument types to match flattened NumPy arrays and calls the CUDA function directly, so existing data-science code stays unchanged.
  • Configurable convolution: the active filter and window size are chosen from a CLI mode string (e.g. edge_n3, sharpen_n7), with zero-padded boundary handling and output clipped to 0-255.

Key findings

  • The GPU outperforms the CPU at every size tested, and the gap widens sharply as matrices grow.
  • Tiling helps most in a mid-size range; at very small sizes its overhead can outweigh the benefit.
  • Our optimized kernel is competitive with - occasionally faster than - cuBLAS on small matrices, while cuBLAS pulls ahead on the heaviest workloads.

Impact

  • 115x speedup over CPU at N=1024 and 14x faster PyTorch preprocessing.
  • GPU-accelerated preprocessing usable from standard Python data-science workflows.
